Claude Code /loop コマンド

理解して現場で使いこなす — セッション限定の定期実行機能を、適切な場面で自信を持って選択できるスキルを身につける

前半: 約10-15分 | 後半: 約25-35分 | ラボ: 約20-30分

このトピックを一言で

/loop は「今の作業セッション中に、指定した間隔でプロンプトやコマンドを繰り返し自動実行する」仕組みです。cron の便利版ではなく、「作業中の一時的な自動監視」に特化した設計です。

あなたが既に知っていること

cron（定期実行のスケジューリング）、ポーリング（定期的な状態確認）、GitHub Actions のスケジュールトリガーなどを知っていれば、/loop は「それらの CLI セッション版」として位置づけることができます。ターミナルで動くプロセスのライフサイクル（プロセス終了 = 全部消える）を理解していれば、/loop の核心がすぐ掴めます。

なぜこれを学ぶのか

デプロイ監視やPRレビューなど、開発中の「ちょっとした定期確認」を手動でやっていませんか？ /loop を正しく理解すれば、コーディングに集中したまま自動的に状況を把握できます。逆に、理解不足で使うと「なぜ翌日消えているの？」という混乱や、不要なAPIコスト、あるいは本来別ツールを使うべき場面での誤用を招きます。

セッション・ライフサイクル図

Section 1 /loop の基本：段階的に理解する

/loop コマンドの実行モデル

定義: /loop [間隔] [プロンプト] で、指定した間隔ごとに同じプロンプトを自動的に再実行するセッション内スケジューラー。デフォルト間隔は10分、最大50タスク登録可、3日で自動満了。

具体例:

$ /loop 5m デプロイのステータスを確認して、エラーがあれば教えて

これにより5分ごとに Claude がデプロイ状況を確認し、問題があればターミナルに通知します。あなたはコーディングを続けたまま、バックグラウンドで監視が行われます。

なぜそうなるか: /loop はセッション（CLIプロセス）に紐づくため、ターミナルを閉じれば全タスクが消えます。これは「忘れられたループが永遠にAPIを叩き続ける」事故を防ぐ意図的な設計です。また、確定的ジッター（task ID に基づく固定オフセット）により、多数のユーザーが同時刻にAPIを呼び出す「サンダリングハード問題」を回避しています。

Mini Lab: 基本動作を体験する

前提条件: Claude Code がインストール済みで、ターミナルから claude コマンドが実行できること。

Basic Exercise 1: 最初の /loop を設定する

最も基本的な /loop コマンドを実行して、動作を観察します。

  1. Claude Code のセッションを開きます
  2. 以下のコマンドを実行します:
claude> /loop 2m 現在時刻を教えて（テスト用: 実行された時刻を表示するだけ）
  1. 実行後、最初の fire が起こるまで待ちます（2分 + ジッター）
  2. fire が起こったら、表示された時刻を確認します
  3. もう1回 fire されるまで待って、間隔を確認します
確認ポイント: 2回の fire の間隔は、正確に2分でしたか？ それとも少しズレがありましたか？ ズレがあったなら、それがジッターです。

このことから何がわかりましたか？

期待される気づき: 実行間隔は「ぴったり2分」ではなく、数秒〜十数秒のズレがある。これが確定的ジッター（task ID に基づく固定オフセット）です。同じタスクは常に同じオフセットを持つため、予測可能ですが正確ではありません。

Intermediate Exercise 4: セッション限定の意味を体験する

/loop の最も重要な制約「セッション限定」を実際に確認します。

注意: この演習では Claude Code セッションを終了します。保存していない作業がある場合は先に保存してください。
  1. /loop が動いている状態を確認します
  2. Claude Code セッションを終了します（/exit または Ctrl+C）
  3. 新しい Claude Code セッションを開きます
  4. 先ほどの /loop タスクがどうなったか確認します
確認ポイント: 以前のタスクは残っていましたか？ 残っていないとしたら、それは「バグ」ですか「設計」ですか？

核心: すべてのタスクが消えているはずです。これは設計上の意図です。/loop は CLI プロセスに紐づいており、プロセスが終了すればタスクも消えます。これにより「忘れられたループが永遠に API を叩き続ける」事故を防いでいます。永続的なスケジューリングが必要なら Desktop scheduled tasks か GitHub Actions を使います。

もう少し複雑な使い方

スキルとの組み合わせ:

# 他のスラッシュコマンドをスケジュール実行
$ /loop 20m /review-pr 1234

# 間隔省略 = デフォルト10分
$ /loop PRに新しいコメントがないか確認して

# 条件付きプロンプト
$ /loop 5m ログにERRORが増えていたら教えて。問題なければ何も出力しないで

ポイント: /loop 自体は「何をするか」に関知しません。プロンプトの内容がすべてを決めます。/loop は「いつ実行するか」だけを担当する薄いラッパーです。

Mini Lab: 複数ループとプロンプト制御

Basic Exercise 2: 複数の /loop を同時に走らせる

2つの /loop を同時に設定して、独立して動作することを確認します。

  1. 新しいセッションで /loop を1つ設定してから、もう1つ追加します:
claude> /loop 3m 現在のディレクトリのファイル数を教えて（テスト用）
  1. 2つのタスクが独立して fire されることを観察します
  2. それぞれの fire タイミングが異なることを確認します
確認ポイント: 2つの /loop は互いに干渉していますか？ 各タスクのジッターは同じですか、異なりますか？

期待される気づき: 各タスクは完全に独立して fire されます。ジッターは task ID ごとに異なる（task ID から決定的に計算されるため）ので、2つのタスクが同時に fire されることは通常ありません。これが「サンダリングハード回避」の仕組みです。

Intermediate Exercise 3: プロンプトで出力を制御する

/loop の出力はプロンプトの書き方で完全に制御できることを体験します。

  1. まず「うるさい」ループを設定します:
claude> /loop 2m このディレクトリの全ファイルの詳細を教えて
  1. 出力が長くて邪魔だと感じたら、同じ監視を「静かに」やり直します:
claude> /loop 2m このディレクトリに新しいファイルが追加されていたら教えて。変化がなければ何も出力しないで
  1. 2つの出力の違いを比較します
確認ポイント: /loop 自体にはフィルタリング機能がありますか？ それとも、プロンプトの書き方が出力を決めていますか？

核心: /loop 自体にはフィルタリング機能は一切ありません。/loop は「いつ実行するか」だけを担当する薄いラッパーです。何を出力するか、どう動作するかは、すべてプロンプトが決めます。これが「プロンプト制御の動作」（設計原則 #7）です。

9つの設計原則

1. セッション限定実行
2. ステートレス（状態なし）
3. 固定間隔スケジューリング
4. 低優先度バックグラウンド実行
5. 確定的ジッター
6. 3日自動満了
7. プロンプト制御の動作
8. スキルとの組み合わせ可
9. 永続化なし・キャッチアップなし

Section 2 暗記者と理解者の違い

同じ質問に対する2つの回答を比較してみましょう。何が違うか考えてください。

Q: /loop でデプロイチェック用のタスクを設定しました。ターミナルを閉じて翌日開いた時、そのタスクはまだ実行されていますか？

暗記者パターン

「/loop で設定した定期タスクは、バックグラウンドで実行されます。cron のように設定すれば、ずっと実行され続けます。」

理解者パターン

「いいえ、実行されていません。/loop はセッション限定で、ターミナルを閉じた時点で全タスクが削除されます。翌日も実行させたければ Desktop scheduled tasks を使うか、/loop を再設定する必要があります。/loop は『今のセッション中の一時的な監視』向けで、3日で自動満了します。」

Q: /loop で最初のチェックが失敗しました。次の実行では、この失敗から学習して修正された動作が期待できますか？

暗記者パターン

「Claude は前の実行結果から学習するので、次は改善されるはずです。コンテキストが保持されるから、試行錯誤で問題を解決できます。」

理解者パターン

「いいえ。/loop は毎回『同じプロンプトを独立して実行』します。前回のエラーから学習する仕組みはありません。Ralph Wiggum（iterative problem-solving）と /loop（periodic observation）は全く異なる概念です。自動修正が必要なら、エラーハンドリングロジックをプロンプト自体に埋め込む必要があります。」

Section 3 よくある誤解と落とし穴

誤解 1: 「/loop は永続的なバックグラウンドタスク管理システムだ」

なぜ間違えるか: 「loop」という言葉が「継続的な反復」を連想させ、cron や systemd のようなシステムレベルのスケジューラーと同一視してしまいます。最初の数分間は実際に動くのを見るので、「設定した = 以降ずっと動く」と帰納してしまうのです。

正しい理解: /loop はセッション内限定のスケジューラーです。ターミナル終了で全タスク即座に削除、同じセッションでも3日で自動満了。永続的なスケジューリングが必要なら Desktop scheduled tasks か GitHub Actions を使います。

誤解 2: 「/loop は前回の結果から学習して、次の実行を改善する」

なぜ間違えるか: 「AIなら学習するはず」という直感的期待と、Claude の会話内での適応能力を /loop にも適用してしまいます。また、Ralph Wiggum（iterative problem-solving）と /loop（periodic observation）を混同してしまいます。

正しい理解: /loop は毎回、独立した同じプロンプトを「盲目的に」実行します。前回のコンテキストは引き継がず、前回の失敗から学習しません。適応型の自動化が必要なら、エージェントフレームワークや iterative パターンを使います。

誤解 3: 「/loop 5m と指定したら、ちょうど5分ごとに実行される」

なぜ間違えるか: 通常の cron の期待値（*/5 * * * * = 正確に5分ごと）をそのまま当てはめてしまいます。数字で指定した interval をそのまま信じる心理もあります。

正しい理解: API 同時アクセス回避のため、確定的ジッター（deterministic jitter）が追加されます。周期の最大10%（上限15分）のオフセットが task ID から決定的に計算されるため、「同じタスク = 常に同じオフセット」ですが、:00秒ぴったりにはなりません。また、ビジー中のミス fire はキャッチアップされません（1回だけ実行）。

Test 5 / 7 What-Ifテスト

条件を変えたらどうなるか？メカニズムに基づいて考えてください（推測ではなく）。

Level 1 — 単一因果

もし /loop が session-scoped ではなく OS レベルで永続化されたら、何が変わりますか？

実装層の変化: タスク保存がセッションメモリから OS ファイルシステム/デーモンに移行。ユーザー側: ターミナル再起動後もタスクが存続し unattended 実行が可能に。3日の自動満了が不要になる。デメリット: 「忘れられたループ」が API を叩き続ける orphaned task 問題が発生。コスト制御が困難になり、memory leak 的な事故リスクが増大します。

Level 2 — 連鎖効果

その場合、Desktop scheduled tasks の存在意義はどう変わりますか？

現在: /loop（短期 CLI）vs Desktop（長期 persistent）で明確に分業。仮定実現時: 両者が機能重複し、Desktop が不要になる可能性。しかし、GUI のアクセシビリティ、セッション間の isolation boundary（credential/MCP アクセスの違い）、CLI スキルがないユーザーへの配慮から、両者は並存し続ける。結果として tool fragmentation（複数ツールの共存で混乱）が起こりえます。

Level 3 — 制約推論

それでも Desktop scheduled tasks が必要とされ続ける技術以外の理由は何ですか？

ユーザー心理: GUI の方が「安全」「可視化される」という印象がある。組織的理由: CLI スキルがない人でもタスク管理できる。技術的残存理由: session と desktop app は異なる isolated environment で、credential/MCP server アクセスが異なる。たとえ /loop が persistent でも、「UI のアクセシビリティ」「isolation boundary」の理由で両者は並存します。

Test 6 / 7 転移テスト

近転移

デプロイの監視に加えて、DBマイグレーションのステータスも5分ごとにチェックしたくなりました。/loop でどう設定しますか？ 2つのアプローチを示し、なぜどちらも機能するかを説明してください。

アプローチ1: プロンプトに両方含める — /loop 5m デプロイとDBマイグレーションの両方のステータスを確認して。アプローチ2: 2つの独立した /loop を作る — /loop 5m デプロイを確認 と /loop 5m DBマイグレーションを確認。なぜどちらも機能するか: /loop はプロンプトの複雑さに関知しません。「いつ実行するか」だけを担当する薄いラッパーなので、プロンプトの内容がシンプルでも複雑でも、スケジューリングのメカニズムは同じです。

中転移

チームは非同期でタイムゾーンをまたいで働いており、PR レビューの自動化を24/7で動かしたい。/loop で実現可能ですか？ 不可能なら何を使うべきか、その理由とともに説明してください。

不可能です。 /loop はセッション限定（ターミナルを閉じたら消える）で、24/7の稼働は保証できません。代替: (1) GitHub Actions の schedule トリガー — GitHub のインフラで実行されるため、ターミナル不要。(2) Desktop scheduled tasks — ローカルで永続的に動作。判断基準: /loop は「monitoring-while-I-work」向け。自動化が restart を跨ぐ必要がある、または unattended で動く必要がある場合、/loop の persistence guarantee では不足します。ツール選択の軸は「タスクの複雑さ」ではなく「persistence requirement（永続性の要件）」です。

遠転移

/loop で health endpoint をポーリングしていますが、トラフィックがバースト的（9時と18時にピーク）で、固定間隔では不十分です。オフピーク時の短い障害を見逃し、ピーク時には不要な負荷をかけています。どう改善しますか？ これは /loop の何の限界を示していますか？

/loop の根本的限界: 「固定間隔のダムポーラー」であること。適応的ポーリング（ピーク時は高頻度、オフピーク時は低頻度）、イベント駆動アラート（webhookで即座に反応）、指数バックオフ（正常なら低頻度、異常なら高頻度）のいずれも /loop では実現できません。解決策: Datadog や Prometheus のような observability プラットフォームへの移行。アナロジー: /loop はメトロノーム（固定テンポで刻む）。適応的な監視はオーケストラの指揮者（テンポを変え、楽器間で同期する）。メトロノームは練習に最適ですが、本番の演奏には指揮者が必要です。

正解と判断基準:

1. デプロイ後30分 → /loop（短期・作業中・セットアップ不要）

2. 毎朝9時の日次レポート → Desktop（永続性が必要・ローカルで十分）

3. 24/7のインフラ監視 → GH Actions（マシン非依存の永続性が必要）

4. コーディング中のPR監視 → /loop（作業セッション中・一時的）

5. 週次セキュリティスキャン → GH Actions（長期間の定期実行・CI/CD統合）

判断の軸: 「タスクの複雑さ」ではなく「persistence requirement（永続性の要件）」と「実行コンテキスト（ローカル or クラウド）」で決まります。

Lab まとめ

Mini Lab と Integration Lab を通じて確認した /loop の核心:

  • セッション限定: ターミナルを閉じたら全タスク消滅（設計上の意図）
  • 確定的ジッター: 実行タイミングは正確ではないが予測可能
  • プロンプトがすべて: /loop 自体は薄いラッパー、動作はプロンプトが決める
  • ツール選択: 永続性の要件で /loop / Desktop / GH Actions を使い分ける

実験チャレンジ

実験 1: ジッター観察

同じインターバルで2つのタスクを作成し、発火タイミングがずれることを確認しよう。
/loop 1m task-alpha → /loop 1m task-beta

実験 2: 3日間の自動期限切れ

スピードを 1s=1day に上げて、3日後にタスクが自動消滅するのを観察しよう。

実験 3: Busy スキップ

タスクを作った後 Make Busy を押し、発火がスキップされ idle に戻ったら1回だけ catch-up されることを確認。

実験 4: 独立タスク

異なるインターバルのタスクを3つ以上作り、それぞれが独立して発火することを確認しよう。

実験 5: セッション終了

複数タスクが動いている状態で End Session → 全タスクが一瞬で消えることを確認。これがセッションスコープ。
