Git Rebase

チーム開発でブランチ履歴を整理できるようになる

前半: 約10-15分 | 後半: 約25-35分

このトピックを一言で

Rebaseとは、ブランチの根っこ（基準点）を別の場所に付け替え、その上でコミット群を順番に再構築する操作です。結果として線形で読みやすい履歴を作りますが、公開済みブランチでは他の開発者の作業を破壊してしまう危険性を持つツールです。

あなたが既に知っていること

Gitコミットの親子関係は家系図のようなものです。Rebaseは「養子に出す」に相当する操作で、血のつながり（パッチ内容）は変わりませんが、法的な親（基準点）が変わり、法的身分（SHA-1ハッシュ）が変わります。Excelで参照元セルを変更すると値が再計算される経験があれば、それと同じ仕組みです。

なぜこれを学ぶのか

チーム開発では、複数人が同時にブランチを切って作業します。その際、履歴が複雑に絡み合うとバグ調査（git bisect）やコードレビューが困難になります。Rebaseを正しく理解すると、ローカルブランチの整理、PRの読みやすさ向上、mainブランチの品質維持を戦略的に設計できるようになります。「いつ使い、いつ使わないか」の判断ができること ― それがこの学習のゴールです。

Section 1 Rebaseの仕組みを段階的に理解する

Example 1: ローカルフィーチャーブランチの整理（安全なRebase）

状況: 開発者AさんがmainブランチからFeature/loginブランチを作成。3個のコミット（認証API実装→UIデザイン→テスト追加）を作成中に、mainブランチが他のメンバーの変更で進展しました。

手順:

1
共通祖先（C2）を特定する
2
A1, A2, A3 のパッチ（差分）を抽出し、一時ファイルに保存
3
featureブランチのポインタをC5（mainの先端）にリセット
4
パッチを順に再適用 → A1', A2', A3' が新しいハッシュで生成される

なぜこうなるか: Gitのコミットハッシュ（SHA-1）は、コミットの内容＋親ポインタ＋メタデータから計算されます。親ポインタがC2→C5に変わるため、同じコード差分でも全く新しいハッシュが生成されます。A1とA1'は「同じ変更内容」だが「異なるコミットオブジェクト」です。

Example 2: 公開ブランチRebaseの破壊的影響

状況: 開発者BさんがmainにコミットXをpush済み。CさんがそのXをpullし、Xに基づいてコミットYを作成。その後、BさんがmainでコミットXをrebaseしてX'に書き換えてしまった。

メカニズム: CさんのコミットYの親ポインタは古いXを指しています。BさんがXをX'にrebase→force pushすると、リモートからXが消え、CさんのYは「存在しない親」を指す孤立状態になります。Cさんがpullすると、同じ変更が異なるハッシュで二重に存在する矛盾が発生します。

核心: 公開済みコミットのハッシュは「社会契約」。それを破ると、そのコミットに基づく全員の作業が破壊されます。これは「気をつければ大丈夫」な問題ではなく、データ構造レベルの矛盾です。

Section 2 暗記者と理解者の違い

同じ質問に対する2つの回答を比較してみましょう。何が違うか考えてください。

Q: git rebaseとは何か、簡潔に説明してください。

暗記者パターン

git rebaseはブランチを別の基点に移動させ、リニアhistoryを作るコマンドです。git mergeと同じ役割ですが、マージコミットを作らずにきれいな履歴を保ちます。

理解者パターン

git rebaseは、現在のブランチの各コミットから差分パッチを抽出し、指定された新しい基点に対して「順番に再適用」していく操作です。各コミットは新しいハッシュを持つ新しいオブジェクトとして生成されます。マージと異なり、分岐の痕跡そのものを歴史から消すため、既に他の開発者に共有されたコミットに対しては使用してはいけません。

Q: git rebaseしたとき、古いコミットはどうなりますか？

暗記者パターン

古いコミットは削除されて、新しい場所に移動します。git reflogで見れば復元できます。

理解者パターン

古いコミットは「削除されない」し「移動もしない」。ブランチポインタが「新しく生成されたコミット群」を指すようになるだけです。rebase後は古いコミットオブジェクトと新しいコミットオブジェクトの両方がリポジトリに存在し、ブランチは後者を指しています。古いコミットに基づいていた他の開発者のブランチがあると、ここで重複が発生します。

この2つの回答の決定的な違いは何だと思いますか？

Section 3 よくある誤解と落とし穴

誤解: 「rebaseは既存のコミットを移動させるだけ」

なぜ間違えるか: ビジュアル表現（図解）が「コミットが枝を移動する」ように見えるため。実際には完全に新しいコミットオブジェクトが生成される仕組みを知らない。

正しい理解: git rebaseは既存のコミットを移動させるのではなく、差分パッチを別の基点に対して「再適用（replay）」し、新しいコミットハッシュを持つ全く新しいコミットオブジェクトを生成します。古いコミットは変わらず存在し、ブランチポインタだけが新しいコミット群を指すようになります。

誤解: 「公開ブランチでrebaseしてはいけないのは慣例」

なぜ間違えるか: ルール自体は知っているが、なぜ「必須ルール」なのか（ハッシュ変更による協調的破損メカニズム）を理解していない。

正しい理解: rebaseで新しいハッシュが生成される → 他の開発者が古いハッシュに基づいて作業している → 重複コミットが発生し、Gitが divergent history と判断 → 強制pushすると全員のコミットが重複する。「気をつければ大丈夫」ではなく、物理的なデータ構造の矛盾です。

誤解: 「rebase -i でrewordしてもコミット自体は変わらない」

なぜ間違えるか: 「編集」という言葉から、元のコミットが変更（modify）されると思い込む。

正しい理解: reword操作でも全く新しいコミットオブジェクトが生成されます。コミットのSHA-1はメッセージも含めて計算されるため、メッセージ変更 = ハッシュ変更 = 新コミット生成です。それ以降の全コミットも連鎖的に新しくなります。

Level 1 — 単一因果

もしあなたがfeature/login-pageでgit rebase mainを実行した後、複数のコンフリクトが出現しました。この時点で「git rebase --abort」を実行すると、何が復元されて、何が復元されないでしょうか？

git rebase --abort実行時点で、rebasing処理の途中で生成された「未完成の新しいコミット群」は削除され、rebase前の「古いブランチポインタ」が復元されます。元々のコミット（rebase前）が復元されるため、コンフリクト前の状態に完全に戻ります。重要なのは「rebase処理中に生成されたコミットはブランチから外れるだけで、オブジェクトとしては削除されない」ということ。reflogから参照できます。

Level 2 — 連鎖効果

上のシナリオでabortせず「コンフリクトを解決してgit rebase --continue」した場合、その後のコミットはどのようなハッシュを持つでしょうか？また、このブランチを既にpush済みだった場合、チーム開発ではどのような問題が起きるでしょうか？

rebase --continue で完了すると、元々のコミット群（古いハッシュ）と新しく生成されたコミット群（新しいハッシュ）が両方リポジトリに存在します。git push -f でリモートを上書きすると、他の開発者のローカルに残った古いハッシュとリモートの新しいハッシュが矛盾し、彼らのpullが失敗します。同じコード変更が異なるハッシュで重複し、historyが破壊されます。

Level 3 — 制約推論

上のチーム開発問題でも「変わらないもの」と「変わるもの」を区別してください。ファイルの内容、コミットハッシュ、コミット数、ブランチの親、他の開発者のコミットのそれぞれについて説明してください。

変わらない: ファイルの最終的な内容は同じ（パッチ内容は保持）。他の開発者が別ブランチで行ったコミットも影響を受けない。
変わる: コミットハッシュが全て新しくなる。ブランチの親チェーンが変わる（元はA→B→C、新しくはmain最新→B'→C'）。
影響を受ける人: このブランチからfork/branch outした人だけ。ローカルで整理するだけなら誰にも影響なし。「誰がこのコミットに依存しているか」が判断基準。

Test 6 / 7 転移テスト

近転移

あなたのfeature/user-authブランチで10個のコミットを作成しましたが、そのうち3つが「デバッグ用」「typo修正」「リンター修正」のコミットです。mainにマージする前に、これらを対応する実装コミットに統合したいとします。どのコマンドを使い、どのような手順で実行しますか？

git rebase -i main を使用します。interactive rebaseで、修正コミットをsquashコマンドで対応する実装コミットに統合します。または、事前に git commit --fixup で修正コミットを作成しておけば、git rebase -i --autosquash main で自動的に整理されます。これは「コミットの論理的なまとまりを保ちながら、技術的なノイズを除去する」という原理の直接的な適用です。

近転移

feature/paymentブランチで作業中、mainブランチに新しい変更が入りました。あなたのブランチをmainの最新に同期したいです（PRはまだ未作成）。手順を説明してください。

(1) git fetch origin で最新のmainを取得、(2) git rebase origin/main で自分のコミットを再適用、(3) コンフリクトがあれば修正して git add → git rebase --continue、(4) git push --force-with-lease で更新。まだ共有していないブランチのため安全です。「ベースとなる土台を最新状態に同期させながら、自分の変更の独立性を保つ」という原理の適用です。

中転移

2つのfeatureブランチ（A: feature/notifications、B: feature/analytics）を開発していて、AはBの上にブランチを切っています。Bの実装に方針変更が入り、Bをmainからrebaseし直しました。Aの状態はどうなり、どう対応しますか？

Aのコミット群はBの古いハッシュに依存しているため、Bのrebase後にAは「孤立した」状態になります。対応：(1) Bをrebase完了後、AをBの新しい先端で git rebase --onto B-new B-old A として再構築、(2) またはAを一度退避してからBの上でrebuild。これは「upstream依存ブランチの変更が下流に波及する」という原理の拡張で、単一ブランチのrebaseから複数ブランチ間の依存関係管理への転移です。

遠転移

新しいウェブアプリプロジェクトで「mainは常に本番対応状態」という原則があります。複数メンバーが毎日機能を追加し、バグ報告時はhotfixとして即日本番反映が必要です。git rebaseとmergeをどう使い分けるポリシーを設計しますか？

推奨ポリシー: (1) feature/*ブランチ：ローカルで頻繁にgit pull --rebaseで最新同期（線形の個人履歴）、(2) PRレビュー中：rebaseしない（履歴の可追跡性のため）、(3) PRマージ時：squash mergeまたはrebase後にmerge（developの履歴を整理）、(4) develop→main：merge --no-ffでマージコミット残す（release trackのため）、(5) hotfix：mainから切り、完了後はmergeでmainとdevelopの両方に統合。「履歴の線形性」と「変更の追跡可能性」のバランスを取り、hotfixの双方向統合という運用制約を組み込んだ設計です。

Basic

Exercise 1: 基本の rebase — 遅れたブランチを最新化

feature/add-farewell は古いコミットから分岐しています。 main が先に進んだので、rebase でブランチの起点を最新の main に移動します。

  1. まず現状を確認する
$ git log --oneline --graph --all $ git checkout feature/add-farewell $ git log --oneline # feature のコミットハッシュをメモしておく
  1. rebase を実行する
$ git rebase main
  1. 結果を確認する
$ git log --oneline --graph --all $ git log --oneline # コミットハッシュが変わっていることを確認
理解チェック: コミットハッシュが変わったのはなぜ？ 変更内容（diff）は同じなのに、ハッシュが異なる理由を「親ポインタ」の観点で考えてみてください。
解説: Git のコミットハッシュ（SHA-1）は、変更内容だけでなく「親コミットのハッシュ」も含めて計算されます。 rebase で親が変わる（古い main → 新しい main の先頭）ため、同じ diff でもハッシュが変わります。 これが「rebase は履歴を書き換える」と言われる理由です。

Exercise 2: interactive rebase — コミットを squash で整理

feature/add-logger には 3 つのコミットがありますが、 最初の 2 つ（logger 追加 + level パラメータ追加）は本来 1 つにまとめるべきです。

  1. ブランチに切り替えて現状確認
$ git checkout feature/add-logger $ git log --oneline aaa1234 feat: add log_error function bbb5678 fix: add level parameter to log ccc9012 feat: add logger module
  1. 直近 3 コミットを interactive rebase
$ git rebase -i HEAD~3 # エディタが開いたら、2行目の "pick" を "squash"（または "s"）に変更： # pick ccc9012 feat: add logger module # squash bbb5678 fix: add level parameter to log ← ここを変更 # pick aaa1234 feat: add log_error function # 保存して閉じる → コミットメッセージを整理して保存
  1. 結果を確認
$ git log --oneline # 3 つ → 2 つのコミットに整理されている
理解チェック: squash は「前のコミットに吸収」します。 では、もし 1 行目（一番古いコミット）を squash にしたらどうなるでしょうか？
解説: エラーになります。squash は「直前のコミットに吸収」する操作なので、 1 行目（最も古いコミット）には「吸収先」がありません。 Git は error: cannot 'squash' without a previous commit と返します。 これは interactive rebase のリストが「古い順」に並んでいることの理解が鍵です。
Basic

Exercise 3: rebase で main に追従（Exercise 2 の続き）

Exercise 2 でコミットを整理した feature/add-logger を、 最新の main に rebase します。

$ git log --oneline --graph feature/add-logger main # まだ古い main から分岐していることを確認 $ git rebase main $ git log --oneline --graph --all # feature/add-logger が main の先頭から生えている
確認: git log --oneline --graph --all で一直線になっていれば成功です。

Exercise 4: コンフリクトを伴う rebase

feature/conflict-demo と main は両方とも config.py を作成していますが、内容が異なります。rebase するとコンフリクトが発生します。

  1. ブランチに切り替えて diff を確認
$ git checkout feature/conflict-demo $ git diff main -- config.py
  1. rebase を実行（コンフリクト発生！）
$ git rebase main CONFLICT (add/add): Merge conflict in config.py error: could not apply xxxxxxx
  1. コンフリクトを解消する
# コンフリクト箇所を確認 $ cat config.py # <<<<<<< HEAD (main の内容) と ======= と >>>>>>> (feature の内容) を確認 # エディタで config.py を開き、両方の変更を統合する # 例: main の命名を採用しつつ MAX_NAME_LENGTH も追加 $ git add config.py $ git rebase --continue # 2つ目のコミットでもコンフリクトが出たら同様に解消
途中でやり直したくなったら: git rebase --abort で rebase 開始前の状態に完全に戻れます。安心して試行錯誤してください。
理解チェック: merge のコンフリクト解消は 1 回で済みますが、 rebase では複数回コンフリクトが発生しました。なぜ？
解説: rebase は各コミットを 1 つずつ順に再適用します。 feature ブランチに 2 つのコミットがあれば、それぞれの再適用時にコンフリクトが起きる可能性があります。 merge は「最終状態同士」を 1 回だけ比較するので、コンフリクト解消も 1 回です。 これは rebase の「コミット単位の再適用」という本質に直結しています。
Compare

Exercise 5: merge と rebase の違いを体感する

同じ操作を merge と rebase の両方でやって、結果の違いを目で確認するのが最も効果的な学び方です。

  1. 比較用のブランチを準備
$ git checkout main # テスト用ブランチを作成 $ git checkout -b experiment/merge-test $ echo "merge test" > merge-test.txt $ git add merge-test.txt && git commit -m "feat: merge test file" # main を進める $ git checkout main $ echo "main progress" > main-progress.txt $ git add main-progress.txt && git commit -m "feat: main progress"
  1. A) merge でやる
$ git checkout experiment/merge-test $ git merge main $ git log --oneline --graph # → マージコミットができ、履歴が分岐→合流する形に
  1. B) rebase でやる（merge を元に戻してから）
$ git reset --hard HEAD~1 # merge を取り消し $ git rebase main $ git log --oneline --graph # → マージコミットなし、一直線の履歴に

比較表

mergerebase
履歴の形分岐 → 合流一直線
コミットハッシュ変わらない変わる（新規コミット）
マージコミット作られる作られない
push 済みでも安全？安全危険
使いどころ共有ブランチの統合ローカルの整理、PR 前の更新
理解チェック: 「rebase の方がきれいだから常に rebase を使うべき」は正しい？ どんな状況で merge を選ぶべきか、そしてその理由を考えてみてください。
解説: 正しくありません。rebase は「履歴を書き換える」ため、 既に push して他の人と共有しているブランチでは使うべきではありません。 他の開発者のローカル履歴と矛盾が生じ、作業が失われる可能性があります。

merge を選ぶべき状況:
• feature ブランチを main に統合する最終段階（PR マージ）
• 複数人で作業している共有ブランチ
• 「いつ、何が統合されたか」の記録を残したいとき

rebase を選ぶべき状況:
• ローカルの feature ブランチを main の最新に追従させる
• PR 前にコミット履歴を整理する（interactive rebase）
• まだ push していない自分だけのコミットを整理する
Advanced

Exercise 6: push 済みブランチの rebase がなぜ危険か

「push 済みのブランチを rebase してはいけない」と言われる理由を、実際に体験します。

  1. ローカルに「リモート」を模倣して push する
$ git checkout main $ git clone --bare . /tmp/rebase-practice-remote.git $ git remote add origin /tmp/rebase-practice-remote.git $ git checkout -b feature/shared-work $ echo "shared work" > shared.txt $ git add shared.txt && git commit -m "feat: shared work" $ git push -u origin feature/shared-work # 「他の人の作業」を模倣: main にコミット追加 $ git checkout main $ echo "someone else's work" > others.txt $ git add others.txt && git commit -m "feat: someone else's work" $ git push origin main
  1. push 済みのブランチを rebase してみる
$ git checkout feature/shared-work $ git rebase main # push しようとすると... $ git push ! [rejected] feature/shared-work -> feature/shared-work (non-fast-forward) error: failed to push some refs
ここで git push --force すると何が起きるか？
リモートの履歴が書き換わり、他の開発者が pull していた旧コミットとの整合性が壊れます。 最悪の場合、他の人の作業が消えます。
  1. 安全な代替手段
# 選択肢A: --force-with-lease（他の人が push していたら拒否される） $ git push --force-with-lease # 選択肢B: rebase を取り消して merge を使う $ git reset --hard origin/feature/shared-work $ git merge main
理解チェック: --force-with-lease は --force とどう違いますか？ 「安全」と言われる理由を考えてみてください。
解説: --force は無条件にリモートを上書きします。 --force-with-lease は「自分が最後に fetch した時点のリモートの状態」を基準に、 他の誰かがその間に push していたら拒否します。

つまり --force-with-lease は「自分だけが作業しているブランチなら安全に force push できる」ための安全装置です。 他の開発者が push した変更を意図せず消してしまうリスクを防げます。

$ rm -rf /tmp/rebase-practice /tmp/rebase-practice-remote.git

使い分けまとめ

rebase を使う場面

  • ローカル feature ブランチを main の最新に追従
  • PR 前にコミット履歴を整理
  • まだ push していないコミットを整理

merge を使う場面

  • feature ブランチを main に統合（PR マージ）
  • 共有ブランチに変更を取り込む
  • 統合の記録を残したいとき
絶対にやらない: push 済みの共有ブランチを rebase / main ブランチを rebase
